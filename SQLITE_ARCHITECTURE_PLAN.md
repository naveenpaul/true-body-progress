# SQLite Architecture Refactor Plan

Created: 2026-04-07
Revised: 2026-04-07 (post eng-review)

## Goal

Stop losing time to recurring SQLite issues by adopting a real data layer instead of patching symptoms.

The diagnosis from the previous draft is correct: database lifecycle, migration logic, and Zustand state are tangled together. The previous prescription was wrong: it tried to invent a service layer, a hand-rolled migration runner, and a hook taxonomy on top of raw SQL — paying the full cost of a refactor without getting the leverage that an ORM provides.

This revision picks one direction and commits: **adopt Drizzle**, fix the lifecycle bug as a side effect, and ship it in 2-3 small PRs with tests.

## Why Drizzle (not raw SQL + custom runner)

The previous plan straddled both options. That was the central confusion. Every benefit listed in §"What Better Projects Do" — schema as source of truth, typed reads, migration history, eliminating hand-maintained `types.ts` — is exactly what Drizzle ships out of the box.

Building a custom migration runner + service layer + query hooks on top of raw `expo-sqlite` reaches the same destination at 5x the code and zero of the type safety. The starter template already on disk at `expo-sqlite-starter-template/` shows the Drizzle path working in Expo SDK 54 today.

Decision: Drizzle. No straddling.

## What we are NOT doing (explicitly out of scope)

- New `client.ts` module alongside `SQLiteProvider` (two init paths is the bug, not the fix)
- Hand-rolled `schema_migrations` table (drizzle-kit handles this)
- "Service layer" between hooks and the DB (React Query + Drizzle is enough at this scale)
- Distinct `use-X-data` / `use-save-X` hook taxonomy (just `useQuery` / `useMutation`)
- `withExclusiveTransactionAsync` migration (single connection, no cross-conn contention to solve)
- Sync, encryption, multi-device, schema versioning beyond `drizzle-kit`
- Touching workouts in early PRs (highest risk, save for last)

## Architecture (target end state)

```
src/lib/db/
  schema.ts              # Drizzle schema — single source of truth
  index.ts               # openDatabaseSync + drizzle() + export db
  migrations/            # generated by drizzle-kit
    0000_baseline.sql    # snapshot of current prod schema
    0001_*.sql           # future changes
    meta/

src/app/_layout.tsx       # useMigrations() → render
                          # no DbBridge, no manual migrate(), no seed bootstrap mixed in
```

Reads/writes go through Drizzle directly from React Query hooks. Stores hold UI state only (workout draft, selected range, modals). No store imports `SQLiteDatabase`.

## PR 1 — Adopt Drizzle (the only PR that matters)

**Goal:** schema as code, generated migrations, baseline snapshot of current prod schema, no data loss for existing users.

### Steps

1. **Install**
   ```
   pnpm add drizzle-orm
   pnpm add -D drizzle-kit babel-plugin-inline-import
   pnpm add expo-drizzle-studio-plugin
   ```
   Update `babel.config.js` to include `inline-import` for `.sql` files (see starter template `babel.config.js`).

2. **Port schema** — translate the 7 tables in `src/lib/db/database.ts` to typed Drizzle definitions in `src/lib/db/schema.ts`. Mechanical work. Preserve every column, every CHECK, every default, every index. Reference: `expo-sqlite-starter-template/db/schema.ts`.

3. **Create `drizzle.config.ts`** at repo root, mirror starter template.

4. **Generate baseline migration**
   ```
   pnpm drizzle-kit generate --name baseline
   ```
   This produces `drizzle/0000_baseline.sql` matching current prod schema exactly. Verify by diffing against `database.ts` SQL.

5. **Replace bootstrap in `src/app/_layout.tsx`**
   - Open DB once: `const expoDb = openDatabaseSync('app.db')`
   - `const db = drizzle(expoDb, { schema })`
   - `useMigrations(db, migrations)` → wait → render
   - Run PRAGMAs (`journal_mode=WAL`, `foreign_keys=ON`) and `seedExercises()` in a `useEffect` after migrations resolve, gated by an "already seeded" check (count exercises; if zero, seed).
   - Delete `DbBridge`. Delete the manual `migrate()` call. Delete the body_metric rebuild block in `database.ts`.

6. **Encode the body_metric nullable fix as migration `0001`** — don't lose it. Generate via:
   ```
   # edit schema.ts to mark weight/waist nullable (already are in current code)
   pnpm drizzle-kit generate --name nullable_body_metric
   ```

7. **Leave repos + stores untouched in this PR.** They keep working — pass them the same `db` handle (now the Drizzle instance, but the raw `expoDb` is still available if any repo needs `getAllAsync`). Goal of PR 1 is *only* to swap the schema/migration substrate, not to rewrite reads.

   Practically: export both `db` (Drizzle) and `expoDb` (raw) from `src/lib/db/index.ts`. Repos that haven't been ported yet use `expoDb` exactly as today.

### Tests (mandatory — this is the regression-risk PR)

Create `src/lib/db/migrations.test.ts`:

1. **Baseline applies cleanly to empty DB** — open `:memory:`, run migrations, assert all 7 tables exist with expected columns via `PRAGMA table_info`.
2. **Migration is idempotent** — run migrations twice on the same DB, assert no error and no duplicate state.
3. **[REGRESSION] body_metric upgrade preserves data** — open `:memory:`, manually create the *old* body_metric table (NOT NULL weight/waist), insert a row, run migrations, assert the row survives and weight/waist are now nullable.
4. **Seed is idempotent** — call `seedExercises` twice, assert exactly 44 exercises.

These four tests are the gate. PR 1 does not land without them green.

### Success criteria for PR 1

- App boots, all features work identically to before
- `_layout.tsx` has one DB init path
- `DbBridge` is gone
- `database.ts` is gone (or shrunk to PRAGMAs + seed only)
- Migration tests pass on `:memory:`
- Drizzle Studio works on device for inspection

**Stop here if you want.** PR 1 alone fixes the lifecycle bug, kills the hand-rolled migration drift, and gives you typed schema + studio. Everything below is incremental polish.

## PR 2 — Port one feature end-to-end as a proof of concept

**Goal:** validate the Drizzle + React Query pattern on the smallest feature before committing to porting everything.

**Pick: profile / `use-user-store`.** Smallest surface, single row, no charts, no transactions.

### Steps

1. Create `src/features/profile/queries.ts`:
   ```ts
   export const userProfileQuery = () =>
     useQuery({
       queryKey: ['user-profile'],
       queryFn: () => db.select().from(userProfile).limit(1).then(r => r[0] ?? null),
     })

   export const useSaveUserProfile = () =>
     useMutation({
       mutationFn: (data) => db.insert(userProfile).values(data).onConflictDoUpdate(...),
       onSuccess: () => queryClient.invalidateQueries({ queryKey: ['user-profile'] }),
     })
   ```
2. Update profile screen to call the hook instead of `useUserStore`.
3. Delete `use-user-store.ts`, `user-repo.ts`.
4. Verify QueryClient is wired in `_layout.tsx` (it should be — React Query is already installed per the previous plan).

### Success criteria

- Profile screen reads/writes through React Query + Drizzle
- No `db` field in any store related to profile
- One screen, one hook, one obvious read path
- Subjective: does this feel better than the old pattern? If yes, continue to PR 3. If no, stop and reassess.

## PR 3 — Port body, dashboard, nutrition

Same pattern as PR 2, applied to body metrics, dashboard (composes user + body + recent sessions + coach), and nutrition. Each as its own commit so any one can be reverted.

**Pattern per feature:**
- `features/X/queries.ts` — query + mutation hooks using Drizzle
- Delete `features/X/use-X-store.ts` (if it only held DB state)
- Or **keep** the store but strip `db`/`setDb` and leave only UI state (chart range, filters, modal flags)
- Delete `lib/db/X-repo.ts`
- Update screens

Body store: keep it for `chartRange` only. Drop everything else.

## PR 4 — Workouts (last, highest risk)

Workouts mix three things and need to be split:

- **Active workout draft** → stays in Zustand (`use-workout-draft-store`). Genuinely client-side state. No DB.
- **Workout history reads** → React Query + Drizzle (`workout-history-queries.ts`).
- **`saveWorkout` mutation** → Drizzle transaction (`db.transaction(async tx => { ... })`). Inserts session + sets atomically. This is the one place transactions actually matter.

### Test (mandatory)

`workout-save.test.ts`: simulate `saveWorkout` failing partway through set inserts on a `:memory:` DB, assert the session is rolled back (no orphan rows).

## Failure modes addressed

1. **Migration runs partway, crashes, reboot.** Drizzle's `useMigrations` writes to its journal table only after each migration completes — partial state cannot mark itself applied. Verified by the idempotency test.
2. **`useMigrations` throws.** Show an error screen in `_layout.tsx` instead of hanging. Add a `migrationError` branch that renders a "Database error — please reinstall" message. Log to console for dev.
3. **Seed runs twice.** Gate seed on `count(exercises) === 0`. Tested.

## What stays in Zustand (after all PRs)

- Active workout draft
- Selected chart range
- Modal open/close flags
- Form draft state

Nothing else. No DB handles, no fetched DB rows, no startup state.

## Anti-patterns to avoid (unchanged from previous draft)

- Storing `SQLiteDatabase` in Zustand
- Calling repos directly from screens
- Hand-editing one giant migration file
- Splitting startup logic across `_layout.tsx`, `database.ts`, and a bridge component
- Inventing service layers when React Query already exists

PR 1 is the only one that's load-bearing. The rest are incremental and can stop at any point if the value isn't there.

## Recommendation: just start with PR 1

Do not start PR 2 until PR 1 is merged and the migration tests are green on a real device build (not just `:memory:`). The migration test on a real device is the only way to catch SQLite version-specific behavior.

## Final call

The previous draft correctly diagnosed that ownership and lifecycle are the problem. It then proposed a solution that perpetuated the same anti-pattern (two init paths, hand-rolled migration drift, parallel type maintenance) under new names.

Drizzle is the boring, proven, in-distribution answer. Adopt it in one focused PR with tests. Everything else is incremental and optional.
